## 一、项目概述

Flow Ecosystem 是一个个人工作流与认知进化系统，通过追踪用户的对话状态（目标对齐、闭环指数、心流深度、认知成长），动态调整 AI 的交互风格，帮助用户在不知不觉中变得更强大。

**核心差异点**：
- 🪞 **认知镜像** —— 只帮用户看见自己的节奏，不做监控机器
- 🎛️ **AI 调音** —— 根据用户状态自动调整语气、深度、推进节奏
- 🔧 **破局指南** —— 工具不是推荐出来的，是从今天的损失里长出来的

## 三、SessionAnalyzer（4维判定系统）

### 3.1 4维定义

| 维度 | 定义 | 判定标准 |
|------|------|----------|
| **目标对齐** | 对话是否围绕用户设定的活跃目标推进 | 高=紧密围绕，中=部分相关，低=偏离或无关 |
| **闭环指数** | 对话是否产生可交付、可验证的产出 | 高=有明确产出，中=有方案但未交付，低=纯讨论 |
| **心流深度** | 用户的注意力投入程度和连续性 | 高=深度沉浸，中=有波动，低=频繁分心 |
| **认知成长** | 对话中是否产生新概念、新连接、新视角 | 高=突破舒适区，中=稳步积累，低=重复已知 |

### 3.2 证据规范

| 层级 | 长度 | 用途 | 生成方式 |
|------|------|------|----------|
| **SessionAnalyzer 输出** | **30字内** | 信号密度，用于内部判定 | LLM 直接输出，正则解析 |
| **/flow_report 展示层** | **50字内** | 叙事密度，用于用户阅读 | LLM 润色层基于 30字扩写 |
| **趋势变化** | 附加 | 每个维度后可附加趋势 | DailyAggregator 计算 |

**关键原则**：30字是"信号密度"（精准、可解析），50字是"叙事密度"（自然、可阅读）。

### 3.3 LLM Prompt 设计

```
你是认知分析师。请按以下步骤分析：

步骤 1：阅读用户的 MEMORY.md，识别当前最活跃的 1-2 个目标
步骤 2：阅读对话原文，判断对话是否围绕这些目标推进
步骤 3：从4个维度给出判定（只选高/中/低），每维附30字内证据

输出格式（严格按此格式，不要 JSON）：
目标感：[高/中/低]，证据：[30字内]
闭环感：[高/中/低]，证据：[30字内]
沉浸感：[高/中/低]，证据：[30字内]
成长感：[高/中/低]，证据：[30字内]
```

### 3.4 解析器设计

**关键决策**：正则兼容 LLM 的多种输出格式
- 兼容中文/英文冒号 `：` / `:`
- 兼容中文/英文逗号 `，` / `,`
- 兼容有无方括号 `[高]` / `高`
- 兼容有无空格分隔
- 兼容换行分隔（无逗号）

**正则表达式**：
```python
pattern = r'(\w+)感[：:](\[?)([高中低])(\]?)[，,]?\s*证据[：:](\[?)([^\]
]+)(\]?)'
```

## 四、StateDistiller（调音系统）

### 4.1 设计哲学

**pace/depth/tone 三维调音**：
- **pace（推进节奏）**：最高优先级，决定 AI 的行为方向
- **depth（讨论深度）**：次优先级，决定内容复杂度
- **tone（语气温度）**：微调层，仅在用户状态差时触发

**不是行业标准，是为"认知进化系统"发明的控制语言**。公式化描述对 AI 是优势——消除 LLM 的解释权，把行为锁死。

### 4.2 12条规则体系

**设计原则**：
- 优先级从上到下，高优先级规则拦截后不再继续匹配
- 崩溃/倦怠层（规则1-3）> 错配/瓶颈层（规则4-8）> 积极状态层（规则9-11）> 兜底（规则12）
- 每条规则的 `description` 融合**客观描述 + 肯定/看见**

| 优先级 | 规则 | 4维条件 | pace | depth | tone | portrait 标签 |
|:---|:---|:---|:---|:---|:---|:---|
| 1 | **卡壳 burnout** | g=`高`, c=`低`, f=`低`, cg=`低` | hold | surface | soft | 卡壳 burnout |
| 2 | **能量耗尽** | `低` 的数量 ≥ 3 | hold | surface | soft | 能量耗尽 |
| 3 | **产出饱和** | g=`高`, c=`高`, f=`低`, cg=`低` | hold | surface | soft | 产出饱和 |
| 4 | **迷失探索** | g=`低`, c=`低`, f=`中/高`, cg=`中/高` | explore | surface | soft | 迷失探索 |
| 5 | **目标漂移** | g=`低`, c=`高/中` | anchor | surface | soft | 目标漂移 |
| 6 | **执行卡壳** | g=`高`, c=`低/中` | converge | deep | neutral | 执行卡壳 |
| 7 | **心流不稳** | f=`低/中`, cg=`高` | explore | surface | neutral | 心流不稳 |
| 8 | **舒适区运转** | c=`高/中`, f=`高/中`, cg=`中/低` | converge | deep | neutral | 舒适区运转 |
| 9 | **四维协同** | g=`高`, c=`高`, f=`高`, cg=`高` | converge | deep | neutral | 四维协同 |
| 10 | **高产出模式** | g=`高`, c=`高`, f=`中/高`, cg=`中` | converge | deep | neutral | 高产出模式 |
| 11 | **认知突破** | cg=`高`, g=`高/中`, c=`中/高`, f=`中/高` | explore | deep | neutral | 认知突破 |
| 12 | **平稳推进** | 其他所有组合 | explore | deep | neutral | 平稳推进 |

### 4.3 portrait 结构

```python
@dataclass
class StatePortrait:
    label: str           # 状态标签（如"执行卡壳"）
    description: str     # 客观+肯定融合描述（≤50字，给 LLM 润色层扩写用）
    suggestion: str      # 可操作建议
    rule_insight: str    # 规则对应阐述（≤100字，解释状态形成机制）
    style: StyleParams   # pace / depth / tone
    tuning_text: str     # 直接写入 MEMORY.md 的调音文本
```

### 4.4 12条规则 rule_insight 全文

| 规则 | rule_insight（≤100字） |
|:---|:---|
| 1 卡壳 burnout | 全面透支不是意志薄弱，是系统过载的信号。此时任何推进都是透支未来，暂停是唯一理性的选择。 |
| 2 能量耗尽 | 多维度低迷通常伴随自我否定，但低迷是神经系统的保护机制，不是缺陷。 |
| 3 产出饱和 | 高产出的代价是认知停滞，这是可持续性的边界。缓冲不是懈怠，是维持长期输出的必要节奏。 |
| 4 迷失探索 | 方向感缺失时，探索本身就是价值。记录假设比锁定答案更重要，试错数据会自然收敛出方向。 |
| 5 目标漂移 | 执行力错配方向，本质是"手段绑架目的"。定期用"这个动作服务主线吗"校准，比加速更重要。 |
| 6 执行卡壳 | 大脑用讨论代替交付，因为讨论的安全感高于交付的不确定性。设定"丑但可用"的标准可破。 |
| 7 心流不稳 | 新想法是认知进化的燃料，但燃料需要容器。先记录不验证，等注意力稳定后再筛选。 |
| 8 舒适区运转 | 熟练执行说明方法已掌握，但掌握即停滞。在舒适区边缘找一个需要学习的新点刻意深入。 |
| 9 四维协同 | 四维同时在线是可复刻的状态，不是运气。记录触发条件（时间/环境/任务类型），建立个人峰值清单。 |
| 10 高产出模式 | 稳定产出是项目推进的基石，但基石之上需要新结构。找一个当前产出中可升级的认知点。 |
| 11 认知突破 | 新连接形成时最脆弱，容易停留在颅内。给一个落地场景，哪怕只是写一段伪代码或画一张图。 |
| 12 平稳推进 | 无卡点本身就是能力，说明系统处于稳态。保持探索，遇到具体信号再切换节奏。 |

### 4.5 tuning 文本格式

**写入 MEMORY.md FLOW_STYLE 锚点的标准格式**：

```markdown
> 当前工作氛围：[pace映射]，[depth映射]，[tone映射]。
> 适合节奏：[suggestion]。
```

**pace 映射**：
- `converge` → 收敛推进
- `explore` → 顺势探索
- `hold` → 静默陪伴
- `anchor` → 锚定回目标

**depth 映射**：
- `deep` → 深度聚焦
- `surface` → 浅层试探

**tone 映射**：
- `neutral` → 中性语气
- `soft` → 温柔语气

### 4.6 MEMORY.md 锚点更新机制

**触发条件**：仅当 `pace` 变化时写入 `FLOW_STYLE` 锚点
- 如果当前 pace = `converge`，新 portrait pace = `converge` → **跳过写入**
- 如果当前 pace = `converge`，新 portrait pace = `explore` → **执行写入**

**显著性阈值设计理由**：
- pace 是最高优先级调音维度（方向性变化）
- depth/tone 变化不触发写入，避免频繁 I/O 和 AI 语气抖动
- pace 不变时，AI 保持当前行为模式，不打扰用户

**容错机制**：
- 读取 `MEMORY.md` 失败 → 返回空字符串，不阻塞主流程
- 写入失败 → 打印日志，不抛异常
- 锚点不存在 → 首次写入时自动创建

## 五、ReportGenerator（报告组装系统）

### 5.1 架构定位

**不是"生成器"，是"组装器"**。真正的智能在 StateDistiller，ReportGenerator 只是把结构化数据翻译成用户能读的格式。

**职责边界**：
| 层级 | 做什么 | 不做什么 |
|------|--------|----------|
| StateDistiller | 判断状态、输出 portrait + tuning | 不组装报告 |
| DailyAggregator | 计算趋势（较昨日提升/下降） | 不生成文案 |
| ReportGenerator | 组装数据 + 调用 LLM 润色 | 不判断状态 |
| LLM 润色层 | 把结构化数据写成自然语言 | 不改变状态判断 |

### 5.2 2次 LLM 调用设计

| 调用 | 目的 | 输入 | 输出 | Token 估算 |
|------|------|------|------|-----------|
| **第1次：portrait 润色** | 把结构化 portrait 写成自然语言 | portrait + 最近3轮对话 | 总结摘要（≤80字）+ 规则洞察（≤60字） | ~500 |
| **第2次：破局指南** | 识别漏洞 + 推荐工具 + 生成行动 | portrait + 对话证据 | 破局指南（纯文本标记格式） | ~800 |

**总消耗：~1300 token/次**，在免费额度范围内完全可控。

### 5.3 portrait 润色 Prompt（铁律约束）

```
你是 Flow 系统的叙事师。你的任务是把结构化数据改写成自然语言，严格禁止添加输入中不存在的信息。

【铁律】
1. 只使用输入数据中的信息，禁止编造任何数字、百分比、价格、统计、历史数据
2. 禁止替换关键词
3. 禁止发挥：不推测、不联想、不补充背景知识、不引入新概念
4. 禁止用"数据显示"、"研究表明"、"历史记录"等话术包装编造内容
5. 语气：中性、克制、不热情
6. 长度：summary ≤80字，rule_insight ≤60字
```

### 5.4 破局指南 Prompt（纯文本标记格式）

**核心原则**：工具不是"推荐"出来的，是从今天的损失里长出来的。

```
你是 Flow 系统破局顾问。

【核心原则】
工具不是"推荐"出来的，是从今天的损失里长出来的。
根据对话中的具体损失，推荐能堵住这个漏洞的真实工具/方法。

【铁律】
1. 问题描述必须包含：时间范围 + 具体行为 + 量化数据
2. 推荐工具必须真实存在，禁止编造工具名。附上来源（如"GitHub: langflow-ai/langflow"）
3. 推荐理由必须解释：这个工具如何具体堵住这个漏洞
4. 节省时间基于对话证据推算：每天类似浪费 × 250工作日 = 年度节省
5. 学习成本基于该工具的实际入门时间估算
6. 禁止用"数据显示"、"研究表明"等话术

【输出格式】严格按以下标记格式，不要JSON：

VULNERABILITY_1: [8-15字问题类型]（[时间范围] [具体行为] [量化数据]）
TOOL_1: [工具名]（[30字内作用描述]）
REASON_1: [推荐理由，30字内]
TIME_SAVED_1: [X] 天/年
LEARNING_COST_1: [X] 小时
SOURCE_1: [工具来源]

VULNERABILITY_2: ...
...

MAX_RETURN: [工具名] → [20字内具体动作，今晚就能做]
QUICK_WIN: [工具名] → [20字内具体动作，今晚就能做]
```

### 5.5 破局指南输出规范（V4）

**A 部分：问题诊断**

| 字段 | 定义 | 格式 |
|------|------|------|
| 优先级 | 🔴1/2/3 | 按损失大小排序 |
| 问题类型 | 8-15字精炼描述 + 证据 | 「问题类型（时间范围 具体行为 量化数据）」 |
| 推荐工具或方法 | 真实工具名 + 作用 | **工具名**（作用描述，30字内） |
| 推荐理由 | 为什么这个工具能堵住漏洞 | 30字内 |
| 节省时间（年） | 年度节省 | X 天/年 |
| 学习成本 | 入门时间 | X 小时 |

**B 部分：开始行动**

| 模块 | 展示规则 |
|------|----------|
| 最大回报工具 | 工具名 + 20字内可开展的动作描述 |
| 即插即用工具 | 工具名 + 20字内可开展的动作描述 |
| 汇总指标 | 年度节省：X 个工作日 \| 学习成本：X 小时 \| 投入回报率：约 X:1 |

**年度节省上限说明**：
> 三项合计 X 天/年，但受限于一年约 250 个工作日，实际可节省上限为 250 天。此处为理论值，实际收益需按用户实际工作天数折算。

3. **「中」档直接显示「中」** —— 不硬归高/低
4. **全时段统一模板** —— 仅数据聚合范围不同
5. **不主动展示** —— 仅用户召唤 `/flow_report` 时输出

## 七、BraveSearch（联网动态工具搜索）

### 7.1 设计原则

**工具不是硬编码的，是从今天的损失里长出来的。**

- BraveSearch 验证工具是否存在（搜 GitHub/官网）

## 八、用户表现好时的 AI 策略

**核心原则**：用户表现好时，AI 最好的策略不是"赞美"，而是**"不打扰 + 帮收成果"**。

| 积极状态 | AI 行为（tuning） | 原因 |
|:---|:---|:---|
| 四维协同 | `converge` + `deep` + `neutral` | 帮用户把峰值产出固化为可交付物 |
| 高产出模式 | `converge` + `deep` + `neutral` | 推进交付，同时提示一个突破点 |
| 认知突破 | `explore` + `deep` + `neutral` | 允许继续探索新连接，但给一个落地场景 |

**展示层（portrait）的肯定策略**：
- 不是"你真棒"，而是"系统看见了" + "这是可复刻的"
- 肯定后紧跟可复刻建议，把情绪转化为行动
